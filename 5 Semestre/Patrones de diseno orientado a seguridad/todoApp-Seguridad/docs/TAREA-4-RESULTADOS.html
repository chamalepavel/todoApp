🔑 Test 1 — Login devuelve accessToken + refreshToken
✓ PASS login devuelve accessToken y refreshToken 1150 ms

Se registra un usuario y luego se hace login. La respuesta debe contener accessToken (JWT con JWT_SECRET, 15min) y refreshToken (JWT con JWT_REFRESH_SECRET, 7d). Ambos deben ser tokens distintos.

// Petición simulada con supertest POST /api/auth/register { email: "test@test.com", password: "Test1234!" } POST /api/auth/login { email: "test@test.com", password: "Test1234!" } // Respuesta obtenida (HTTP 200) { "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "user": { "id": "...", "email": "test@test.com", "role": "user" } }

Assertions verificadas:

✓ res.statusCode === 200
✓ res.body.accessToken está definido
✓ res.body.refreshToken está definido
✓ res.body.user.email === "test@test.com"
🛡️ Test 2 — Access Token permite acceder a /tareas
✓ PASS access token permite acceder a tareas 769 ms

El access token obtenido del login se usa como Bearer token en el header Authorization para acceder al endpoint protegido /api/tareas. El middleware authenticateToken valida con JWT_SECRET.

// Petición simulada con supertest GET /api/tareas Authorization: Bearer <accessToken> // Respuesta obtenida (HTTP 200) [ ] // Array de tareas del usuario

Assertions verificadas:

✓ res.statusCode === 200
🔄 Test 3 — Refresh genera nuevos tokens
✓ PASS refresh genera nuevos tokens 738 ms

Se envía el refreshToken del login al endpoint POST /api/auth/refresh. La respuesta debe contener un nuevo accessToken y un nuevo refreshToken diferente al original (rotación de tokens).

// Petición simulada con supertest POST /api/auth/refresh { "refreshToken": "<refreshToken-del-login>" } // Respuesta obtenida (HTTP 200) { "accessToken": "eyJhbGci... (NUEVO access token)", "refreshToken": "eyJhbGci... (NUEVO refresh token diferente)" }

Assertions verificadas:

✓ res.statusCode === 200
✓ res.body.accessToken está definido
✓ res.body.refreshToken está definido
✓ res.body.refreshToken !== refreshToken original (rotación confirmada)
🚫 Test 4 — Refresh token viejo es rechazado (Rotación)
✓ PASS refresh token viejo es rechazado despues de rotar 733 ms

Después de hacer un refresh exitoso (Test 3), el refresh token anterior ya no existe en el store (fue eliminado). Intentar usarlo de nuevo debe retornar un error 401.

// Paso 1: Rotar el token (consume el refreshToken viejo) POST /api/auth/refresh { refreshToken: "<refreshToken-ORIGINAL>" } → HTTP 200 — genera nuevos tokens, elimina el viejo del store // Paso 2: Intentar usar el mismo token viejo POST /api/auth/refresh { refreshToken: "<refreshToken-ORIGINAL>" } // Respuesta obtenida (HTTP 401) { "error": "Invalid or revoked refresh token" }

Assertions verificadas:

✓ res.statusCode === 401
✓ res.body.error === "Invalid or revoked refresh token"
🚪 Test 5 — Logout invalida el refresh token
✓ PASS logout invalida el refresh token 598 ms

Al llamar a POST /api/auth/logout con el refreshToken, éste se elimina del store en memoria (Map). Cualquier intento posterior de usar ese token para hacer refresh debe ser rechazado con 401.

// Paso 1: Logout — invalida el refreshToken POST /api/auth/logout { refreshToken: "<refreshToken-actual>" } // Respuesta obtenida (HTTP 200) { "message": "Logged out successfully" } // Paso 2: Intentar refresh después del logout POST /api/auth/refresh { refreshToken: "<refreshToken-del-logout>" } // Respuesta obtenida (HTTP 401) { "error": "Invalid or revoked refresh token" }

Assertions verificadas:

✓ logout.body.message === "Logged out successfully"
✓ res.statusCode === 401
✓ res.body.error === "Invalid or revoked refresh token"
